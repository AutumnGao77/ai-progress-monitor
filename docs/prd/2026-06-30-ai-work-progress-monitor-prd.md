# AI 工作进度监控桌面助手 PRD

> 历史基线：本文件记录 2026-06-30 阶段的早期产品设想。当前执行目标以 `2026-07-01-ai-sloth-pet-monitor-ai-coding-prd.md` 为准；新版已取消宠物内 Yes/No 主路径，改为原创树懒 Pet、气泡列表、点击回原窗口、右键隐藏/退出、通用 AI 工具配置和桌面端会话收口规则。

## 1. 结论

本产品建议做成一个 **跨平台、低侵入的悬浮桌面宠物式监控助手**：常驻桌面但不占用过多画面，全面监控 Claude Code 与 Codex 在终端和桌面端中的工作状态。当任一会话进入“需要用户处理”状态时，在 5 秒内给出明显但不过度打扰的提醒。若只是低风险 Yes/No 确认，用户可直接在宠物中处理；若需要阅读、输入或多轮交互，则引导用户快速定位到对应窗口。

## 2. 背景与痛点

用户在日常工作中会同时开启多个 Claude Code 和 Codex 会话，这些会话可能运行在终端，也可能运行在桌面端。当用户切到其他页面处理工作时，无法及时知道各个 AI 编程会话的进度，导致频繁切回窗口查看。

| 当前痛点 | 影响 |
|---|---|
| 不知道哪个会话正在执行 | 需要反复切窗口确认 |
| 不知道哪个会话需要回复、确认或点击 | 容易错过人工介入时机 |
| 多个 Claude Code / Codex 窗口状态混杂 | 上下文切换成本高 |
| 执行完成或卡住后没有统一提醒 | 工作节奏被打断 |
| 不同系统、不同工具形态不统一 | 难以分享给其他人使用 |
| 简单 Yes/No 确认也要切回窗口 | 小操作被放大成上下文切换 |
| 正在一个 AI 窗口交互时看不到其他窗口进度 | 容易忽略并行任务的处理点 |
| 常驻工具容易遮挡工作区域 | 影响文档、浏览器、终端等正常操作 |

## 3. 产品目标

| 目标 | 说明 |
|---|---|
| 降低主动查看频率 | 用户无需频繁切回 Claude Code 或 Codex 页面 |
| 及时发现待处理事项 | 需要用户介入时主动提醒 |
| 一眼理解整体状态 | 展示所有被监控会话的简要状态 |
| 快速回到对应现场 | 提醒后可定位到对应窗口或会话 |
| 支持通用桌面系统 | 产品可在 macOS 和 Windows 上被其他用户使用 |
| 减少低风险切换 | 对简单 Yes/No 确认支持宠物内处理 |
| 并行状态持续可见 | 即使用户正在某个 AI 窗口中操作，也持续显示其他窗口进度 |
| 低侵入常驻 | 默认不起眼、不遮挡、不抢焦点，用户想看时随时能看懂 |

## 4. 目标用户

主要用户是同时使用 Claude Code、Codex 等 AI 编程工具进行多任务开发、需求验证或文档生成的人。第一版以单人本地使用为主，但产品形态需要具备通用性，后续可以被其他 macOS 和 Windows 用户安装使用。

## 5. 平台与工具支持范围

产品需要同时考虑“AI 工具”和“运行形态”两个维度，避免只适配单一窗口或单一系统。

| 工具 | 终端形态 | 桌面端形态 | 支持要求 |
|---|---|---|---|
| Claude Code | 支持 | 如存在桌面承载形态则支持窗口级监控 | 全面支持 |
| Codex | 支持 | 支持 Codex 桌面端 | 全面支持 |
| 其他 AI 编程工具 | 后续扩展 | 后续扩展 | 预留接口 |

| 操作系统 | 支持要求 | 说明 |
|---|---|---|
| macOS | 必须支持 | 覆盖常见终端、桌面端窗口、系统通知 |
| Windows | 必须支持 | 覆盖常见终端、桌面端窗口、系统通知 |
| Linux | 暂不作为第一优先级 | 架构预留，但不影响 macOS / Windows 交付 |

## 6. MVP 范围

| 模块 | 第一版能力 | 不做 |
|---|---|---|
| 悬浮宠物 | 常驻桌面，表达全局状态 | 复杂换装、成长系统 |
| 低侵入显示 | 默认小尺寸、边缘停靠、半透明或弱提示 | 大面积面板常驻遮挡 |
| 状态监控 | 识别 Claude Code / Codex 在终端和桌面端中的执行中、空闲、需要处理、异常/疑似卡住 | 深度理解完整任务内容 |
| 状态列表 | 点击宠物展开所有会话概览 | 复杂项目管理看板 |
| 提醒 | 对“需要处理”状态做可见提醒 | 高频推送、复杂自动化 |
| 快速定位 | 帮助用户回到对应窗口 | 自动替用户回复或点击 |
| 轻量确认 | 对明确的 Yes/No、Allow/Deny 类操作支持宠物内点击 | 大段输入、多轮阅读、复杂选择 |
| 跨平台支持 | 同时面向 macOS 和 Windows 设计 | 只服务单一电脑环境 |

## 7. 核心用户场景

| 场景 | 用户行为 | 产品响应 |
|---|---|---|
| 多个任务并行执行 | 用户切到浏览器、文档或会议软件 | 宠物保持低打扰状态，显示“执行中” |
| 某个 Claude Code 需要确认 | 用户不在终端窗口 | 宠物变为提醒状态，并标出对应会话 |
| Codex 等待用户回复 | 用户正在其他页面工作 | 弹出轻提示，点击后定位到 Codex |
| Claude Code 只询问是否继续 | 用户看到宠物提醒 | 用户可在宠物内点击 Yes 或 No，无需切回终端 |
| Codex 需要阅读长段上下文后决策 | 用户看到宠物提醒 | 宠物提示“需进入窗口处理”，不在宠物内代替交互 |
| Windows 用户同时运行多个终端会话 | 用户在其他应用中工作 | 产品用同一套状态语言展示所有会话 |
| macOS 用户同时使用桌面端和终端 | 用户不想逐个切窗口查看 | 产品聚合所有 Claude Code / Codex 状态 |
| 用户正在一个 AI 窗口中沟通 | 其他 Claude Code / Codex 会话发生状态变化 | 宠物继续展示其他会话进度并提醒 |
| 用户正在写文档或看网页 | 宠物常驻在屏幕边缘 | 不遮挡主要内容，仅用小状态变化表达进度 |
| 某个任务长时间无变化 | 用户未察觉任务可能卡住 | 标记为“疑似卡住”，但提醒优先级低于“需要处理” |

## 8. 状态定义

| 状态 | 含义 | 提醒优先级 |
|---|---|---|
| 执行中 | AI 正在输出、运行命令或等待工具完成 | 低 |
| 需要处理 | 需要用户回复、确认、点击授权或继续操作 | 高 |
| 空闲/完成 | 当前会话没有继续动作，可能已完成 | 中 |
| 疑似卡住 | 长时间无输出、无状态变化，且未明确完成 | 中低 |
| 无法识别 | 暂时无法判断窗口状态 | 低 |

## 9. 可在宠物内处理的交互边界

宠物内操作只处理低风险、低信息量、明确按钮式的交互，避免误操作。

| 交互类型 | 是否支持宠物内处理 | 规则 |
|---|---|---|
| Yes / No | 支持 | 问题明确，选项只有确认或拒绝 |
| Allow / Deny | 支持 | 操作含义清楚，需展示来源会话 |
| Continue / Stop | 支持 | 不涉及大段上下文判断 |
| 多选项决策 | 暂不支持 | 引导用户回到原窗口 |
| 大段阅读后回复 | 不支持 | 引导用户回到原窗口 |
| 自由文本输入 | 不支持 | 引导用户回到原窗口 |
| 涉及敏感命令或高风险操作 | 不支持 | 必须回到原窗口确认 |

## 10. 提醒策略

第一版提醒应克制，优先解决“错过处理点”的问题。

| 触发条件 | 提醒方式 |
|---|---|
| 出现“需要处理”状态 | 宠物变为明显提醒状态，并显示会话名称 |
| 多个会话同时需要处理 | 汇总提醒，按更新时间倒序 |
| 需要简单 Yes/No 确认 | 宠物展示来源会话、问题摘要、Yes/No 按钮 |
| 需要大量阅读或输入 | 宠物提示“需进入窗口处理”，点击定位到窗口 |
| 状态从执行中变为空闲/完成 | 轻提示，不强打断 |
| 疑似卡住 | 延迟提醒，避免误报 |

## 11. 信息展示

悬浮宠物默认只展示整体状态，点击后展开简版面板。

| 信息 | 示例 |
|---|---|
| 工具类型 | Claude Code / Codex |
| 运行形态 | 终端 / 桌面端 |
| 会话名称 | `pricing-page-refactor` |
| 当前状态 | 需要处理 |
| 状态变化时间 | 2026-06-30 10:58 |
| 简短提示 | “等待用户确认命令” |
| 可操作按钮 | Yes / No，仅在低风险确认时展示 |

默认不展示完整终端内容或大段对话内容，避免泄露敏感信息。

## 12. 低侵入体验要求

产品必须在“可感知”和“不打扰”之间取得平衡。

| 要求 | 说明 |
|---|---|
| 默认小尺寸 | 常驻态只占屏幕很小区域，不影响主要工作窗口 |
| 边缘停靠 | 默认靠近屏幕边缘或角落，减少遮挡正文、按钮、输入框 |
| 不抢焦点 | 状态变化不自动抢占键盘输入焦点 |
| 可快速展开 | 用户点击或悬停后再展示详细状态列表 |
| 可拖动 | 用户可把宠物移动到不影响当前工作的地方 |
| 可临时隐藏 | 用户可短时间隐藏或静音提醒 |
| 分级提醒 | 执行中弱提示，需要处理强提示，避免所有状态都打扰 |
| 一眼可读 | 用户主动看宠物时，能立刻知道是否有任务需要处理 |

## 13. 成功指标与验收标准

| 指标 | 验收标准 |
|---|---|
| 提醒及时性 | 会话进入“需要处理”后 5 秒内提醒 |
| 状态可理解性 | 用户 3 秒内能判断是否需要介入 |
| 误打扰控制 | 执行中状态不反复弹窗 |
| 多会话支持 | 第一版至少支持同时展示 3 个以上会话 |
| 工具覆盖 | 支持 Claude Code 与 Codex 的终端和桌面端监控 |
| 系统覆盖 | 支持 macOS 与 Windows 的基础安装和使用 |
| 轻量确认 | 明确 Yes/No 类确认可在宠物内完成 |
| 交互边界 | 复杂输入、阅读、多选项、高风险操作必须引导回原窗口 |
| 并行监控 | 用户正在一个 Claude/Codex 窗口中操作时，其他窗口状态仍正常展示和提醒 |
| 低侵入体验 | 常驻态不遮挡主要工作内容，不自动抢焦点 |
| 隐私保护 | 默认不上传、不保存敏感会话全文 |

## 14. 隐私与安全要求

| 要求 | 说明 |
|---|---|
| 本地优先 | 第一版仅做本地状态监控 |
| 最小读取 | 只读取判断状态所需的信息 |
| 默认脱敏 | 不在界面长期展示敏感内容 |
| 用户可控 | 用户可以暂停监控或关闭提醒 |
| 操作可追溯 | 宠物内点击 Yes/No 后记录本地操作时间、来源会话和结果 |

## 15. 已确认默认策略

以下问题按默认建议执行，后续进入技术方案阶段验证可行性。

| 策略项 | 默认决策 |
|---|---|
| macOS 与 Windows 支持 | 产品要求同步支持，工程上可分阶段验证 |
| 终端状态读取 | 优先探索终端窗口标题、进程、输出状态 |
| 桌面端状态读取 | 先做窗口级识别，再评估更深集成 |
| 终端软件覆盖 | 优先支持系统默认终端与主流终端 |
| 声音提醒 | 默认关闭，仅视觉提醒 |
| 会话命名 | 支持自动识别，允许用户改名 |
| 默认展示形态 | 小尺寸、边缘停靠、低侵入常驻 |

## 16. 非目标

第一版不做自动回复 AI、不替用户处理复杂任务、不上传工作内容、不做团队协作、不做复杂任务管理系统，不做大面积常驻看板。产品重点是提醒用户“什么时候该回来看”，并在低风险 Yes/No 场景减少不必要切换，而不是接管 Claude Code 或 Codex 的工作。
